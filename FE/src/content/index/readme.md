# FE/src/content/index 组件接入说明

本文档用于规范在 `FE/src/content/index/` 下引入 ReactBits（或其他第三方交互组件）时的使用方式，避免出现“鼠标滚轮失效、页面无法滚动、交互冲突”等问题。

## 本次问题结论（ScrollStack）

### 现象
在 `index.mdx` 引入 `ScrollStack` 后，鼠标滚轮在组件区域内失效或卡住。

### 原因
`ScrollStack` 默认是**局部容器滚动模式**（`useWindowScroll=false`），组件内部使用 Lenis 接管滚动。页面本身还有全局滚动管理时，会出现滚动事件争用，导致滚轮体验异常。

### 当前修复
在 `index.mdx` 使用组件时显式开启窗口滚动：

```mdx
<ScrollStack client:load className="h-[560px]" useWindowScroll={true}>
  ...
</ScrollStack>
```

## 接入第三方组件标准流程

### 1) 先看“是否接管滚动”
接入前先确认组件是否包含以下能力：
- Lenis / Locomotive Scroll / ScrollTrigger / 自定义 wheel 监听
- 强制 `overflow: hidden`
- 局部滚动容器（`overflow-y-auto` + 固定高度）

如果有以上任一项，必须先做滚动策略设计（见第 3 条）。

### 2) 先本地隔离验证
先单独放入一个最小 Demo 区块验证，再接入首页完整内容，避免一次性混入多个组件导致难排查。

### 3) 滚动策略必须二选一
一个页面只建议一种主滚动策略：
- **窗口滚动优先（推荐）**：组件跟随页面滚动（如 `useWindowScroll={true}`）
- **局部滚动**：仅当业务明确需要容器内部独立滚动时使用

不要在同一页面同时让多个库接管滚动。

### 4) Astro 客户端指令选择
涉及滚动监听或 DOM 测量的组件，优先使用：
- `client:load`

避免在关键首屏滚动组件上使用过晚激活策略，防止初始化时机导致高度/偏移计算错误。

### 5) 容器尺寸要可预期
- 避免不必要的固定高度
- 如必须固定高度，确认内容高度、padding、spacer 不会导致滚动边界异常
- 检查 `min-h-screen`、`pb-*` 等是否改变滚动终点

### 6) 重点检查 CSS 冲突
接入后检查是否新增或触发以下风险样式：
- `overflow: hidden`
- `overscroll-behavior: contain`
- `position: fixed` 全屏遮罩
- `pointer-events` 异常设置

### 7) 移动端一起验
除鼠标滚轮外，至少验证：
- 触摸滑动是否顺畅
- iOS/Android 是否存在卡住或反向滚动

### 8) 控制台与清理
- 看是否有 hydration 警告或运行时报错
- 组件卸载时确保销毁动画帧、事件监听、滚动实例

## 建议的接入检查清单（每次引入都过一遍）

- [ ] 组件是否接管滚动（是/否）
- [ ] 页面是否已有其他滚动管理
- [ ] 本页选择了唯一滚动策略
- [ ] 组件使用了合适的 `client:*` 指令
- [ ] 在组件区域内滚轮连续可用
- [ ] 页面其他区域滚轮连续可用
- [ ] 移动端触摸滚动可用
- [ ] 控制台无新增错误

---

## 当再次出现滚轮失效时

按顺序排查：
1. 把组件临时切到窗口滚动模式（若支持）
2. 去掉组件外层固定高度，观察是否恢复
3. 关闭/移除局部 `overscroll-behavior: contain`
4. 检查是否同时启用了两个滚动库
5. 用最小 Demo 回归，逐个恢复功能定位冲突点
